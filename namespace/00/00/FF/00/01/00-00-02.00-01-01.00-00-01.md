```
node_id: 00-00-02.00-01-01.00-00-01
node_nam: Node Identifier
node_tag: NODE_ID
```

## Rules

- A syntactically valid [NODE_ID] denotes one position in [FROID_NS], whether assigned or unassigned.
- Parsers MUST accept lowercase hexadecimal input and normalize it to canonical uppercase form.
- Two [NODE_ID] are equal iff their canonical forms are byte-identical.
- Only an assigned [NODE_ID] identifies a [FOLIO].
- An unassigned [NODE_ID] denotes an empty namespace position.
- When lower [SEGMENT] values are `00`, the assigned [NODE_ID] identifies the [FIN] at the depth of its deepest non-zero [SEGMENT].
- A canonical [NODE_ID] can be brought into alignment with [X660] by the deterministic forward mapping defined in the X.660 alignment spec.
- FROID internally uses [NODE_ID] for all references. [NODE_TAG] and [NODE_NAM] are human-facing layers that resolve to [NODE_ID].
- [NODE_TAG] is the concise, human-parseable short identifier for a [NODE_ID]. It is always written enclosed in [BRACKET]. It maps 1:1 with [NODE_ID] when assigned.
- [NODE_NAM] is the descriptive human-readable label for a [NODE_ID], used in explanatory prose. It maps 1:1 with [NODE_ID]. It is not a reference form and does not appear inside [BRACKET].
- Every system [NODE_ID] receives a [NODE_TAG]. No exceptions for nodes in the system namespace.
- [NODE_ID] entries in user trees may receive a user-defined [NODE_TAG] but are not required to. A user [NODE_ID] without a [NODE_TAG] remains fully addressable by coordinate.
- System [NODE_TAG] values are ALL CAPS, eight characters or fewer, and use only the character set [A-Z][0-9][_]. They are always written enclosed in [BRACKET].
- User [NODE_TAG] values are CamelCase, have no character limit, and are always written enclosed in [Bracket]. User tags are valid only within the user's own RR.
- A bracketed reference in [ALL_CAPS] form is a system [NODE_TAG]. A bracketed reference in [CamelCase] form is a user [NODE_TAG]. The visual register is a reliable signal of which namespace the reference resolves under.
- A [NODE_TAG] that names a field within another concept (such as [STAGING] within [FLIO_REF]) is allocated as a child of the parent concept's [NODE_ID]. Field tags do not occupy peer-level coordinates with the concepts that contain them.
- A [NODE_ID] receives a [NODE_TAG] when it is referenced in human-or-agent-readable content. Internal implementation structures that are not referenced do not require [NODE_TAG] allocation.
- [NODE_ID] identifies [FOLIO] instances only. Artifact filenames that are [NODE_ID]-shaped reference strings are not [NODE_ID] and do not by themselves identify [FOLIO].
- Controlled-vocabulary [SEGMENT] (such as [RR] and [MM]) retain their assigned vocabularies at any [SEGMENT] width; wider encoding space does not extend or relax those vocabularies.
- X.660 forward mapping converts each [SEGMENT] to its decimal arc value independent of [SEGMENT] width; arc values may exceed 255 when [SEGMENT] width exceeds two hexadecimal digits.
- System [NODE_TAG] character count is a visual depth signal: shorter tags identify higher position nodes (typically 4 characters at concept/CC level), longer tags identify deeper nodes (8+ characters at UU and beyond). This convention MUST be applied consistently within each position tier; mixed-length tags at the same depth break the visual parsing convention and complicate reading.

## Constraints

- Each [NODE_ID] consists of exactly nine [SEGMENT], each two or more hexadecimal digits. All [SEGMENT] within a single [NODE_ID] MUST have the same width. The [SEGMENT] width is fixed for the entire [FROID_NS] at instantiation and does not change.
- `00` in a [SEGMENT] denotes the identity position at that position.
- `FF` is reserved: syntactically valid in a [NODE_ID] string but MUST NOT appear in an assigned [NODE_ID].
- Values `01` to `FE` are assignable unless the position document further constrains them.

## Relations

- [NODE_ID] is composed of nine [SEGMENT].
- [NODE_ID] identifies one position in [FROID_NS].
- [NODE_ID] identifies exactly one [FOLIO] when assigned.
- [NODE_ID] has a human-readable name [NODE_NAM] and a short name [NODE_TAG].

## Notes

- Example normalization: `00.01.02.00.00.03.12.01.b5` → `00.01.02.00.00.03.12.01.B5`
