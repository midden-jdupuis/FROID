## Definition

[PROJ_PRG] is the [PROJECT] for software development structure. A [PROJ_PRG] [NODE_ID] addresses a code element — a unit, a callable, or an internal — within a software system.

## Rules

- [PROJ_PRG] declares 3 [NODE_BLK]:
  - Block 1 (WHO): VV / BB / MM (framework-fixed).
  - Block 2 (WHAT NAMESPACE): RR / CC / TT — RR is tenant-allocated; CC=03 ([PROJ_PRG]'s catalog-row id); TT classifies against [TYPE].
  - Block 3 (WHAT ELEMENT): UU / KK / II — UU is the unit segment; KK is the callable segment; II classifies against the internal-element axis.
- **UU (unit)** identifies a language-agnostic unit: a file, module, namespace, or source unit within a [PROJ_PRG] instance.
- **KK (callable)** identifies a function, routine, method, or macro within a unit.
- **II (internal)** is the [SEG_BINN] terminating block 3, classifying internal elements (parameters, locals, types, etc.) attached to a callable.
- A [PROJ_PRG] [NODE_ID] with deeper segments at zero addresses a coarser-resolution position: UU=0 the project root; UU=N KK=0 the unit; UU=N KK=M the callable; UU=N KK=M II=K an internal element on the callable.

## Constraints

- Block 3 segment functions are [PROJ_PRG]-specific. Other [PROJECT] declare different block-3 layouts.
- The internal-element classification axis (II values) is open in v1.0; specific [SEG_BINN] entries may be authored in [PROJ_PRG]'s subtree (under UU=00 KK=00 II=N) when use cases require.
- TT must classify against [TYPE]. v1.0 [PROJ_PRG] uses TT to identify artifact types stored at the addressed position (e.g., source file types).

## Relations

- [PROJ_PRG] is a child of [PROJECT].
- [PROJ_PRG] is applied wherever an (VV, BB) tree assigns RR to a [PROJ_PRG] instance; the applied [NODE_ID] carries CC=03.

## Notes

- Worked example layout. Under FROID, Inc. at RR=01: a callable `parse_node_id` in unit `node_id.py` would have a [NODE_ID] of the shape `01-01-00.01-03-TT.UU-KK-II`, where UU is `node_id.py`'s unit-id within the project, KK is `parse_node_id`'s callable-id within the unit, and II=00 (the callable itself, no internal element).
- [PROJ_PRG] is the projection underlying the FROID, Inc. tenant tree's software work; its block-3 segment roles informed the substrate's chapter layout under [PROJ_FRD], where UU=unit/KK=callable/II=internal-bin labels were carried over for self-instantiation. Other tenants writing software under their own (VV, BB) instantiate [PROJ_PRG] independently.
- A [PROJECT] needing to scope [PROJ_PRG] code addresses (e.g., scoping by cog or container) extends [PROJ_PRG] via prepend-scope: a new [PROJECT] catalog entry prepends a scoping [NODE_BLK] (see [PROJECT] Notes for the extension rule).
