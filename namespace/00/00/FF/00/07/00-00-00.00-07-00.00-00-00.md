## Definition

[PROJECT] specifies one projection of [FROID_NS]. Each [PROJECT] declares its [NODE_BLK] count and authors the segment functions for blocks 3..N. Block 1 (WHO) and Block 2 (WHAT NAMESPACE) are framework-fixed across all [PROJECT].

## Rules

- Every (VV, BB) tree instantiates exactly one [PROJECT] per RR slot.
- Block 1 (WHO) is framework-fixed across all [PROJECT]: VV (position 1, company — framework-fixed role) / BB (position 2, branch — tenant-internal, the company allocates its branches) / MM (position 3, [SEG_BINN] classifying against [MODE]).
- Block 2 (WHAT NAMESPACE) is framework-fixed across all [PROJECT]: RR (position 4, instantiation — tenant-allocated, selects which [PROJECT] applies) / CC (position 5, projection-id — equals the [PROJECT]'s catalog-row id) / TT (position 6, [SEG_BINN] classifying against [TYPE]).
- CC carries the [PROJECT]'s catalog-row id. A [PROJECT] entered at UU=N in this catalog carries CC=N wherever it is applied. Catalog row position and applied CC value are the same number, across all instantiations and all (VV, BB) trees.
- Project-authored [NODE_SEG] begin at UU (block 3, position 7). [PROJECT] specifies the segment functions and bin classifications for blocks 3..N.
- [PROJECT] is fixed at instantiation. Block count and segment functions do not mutate within instantiated [PROJECT]. Cross-projection variation under one (VV, BB) tree uses different RR; cross-company variation uses different VV.

## Constraints

- [FROID_NS] root sits at (VV=00, BB=00) and admits multiple tenant trees. FROID, Inc. occupies (VV=01, BB=01) and currently instantiates one [PROJECT] at RR=01 (software platform bootstrap). Other [PROJECT] instantiations under (VV=01, BB=01) are reserved at higher RR and not yet materialized.
- [PROJECT] MAY NOT redefine framework-fixed segment functions: VV (position 1, company), BB (position 2, branch), MM (position 3, [MODE] bin), RR (position 4, instantiation), CC (position 5, projection-id), TT (position 6, [TYPE] bin).
- CC=00 is reserved (identity / no-projection-applied). Catalog rows enumerate from CC=01.

## Relations

- [PROJECT] is child of [FROID_NS].
- [PROJECT] composes [NODE_BLK] into [NODE_ID] structure for blocks 3..N.
- RR instantiates [PROJECT] in every [NODE_ID] at block 2, position 4.
- CC carries [PROJECT]'s catalog-row id in every [NODE_ID] at block 2, position 5.

## Notes

- A new [PROJECT] MAY **extend** an existing [PROJECT] by prepending a [NODE_BLK] that scopes the blocks following it. The extending [NODE_ID] reads `{project frame}.{extended block}.{parent blocks}` — the framework-fixed blocks 1-2, then the prepended scoping [NODE_BLK] (this [PROJECT]'s contribution), then the parent [PROJECT]'s project-authored blocks. Extensions stack: each extension layer prepends one [NODE_BLK]. The extending [PROJECT] receives its own catalog row and CC value; "extends" is the prose convention used in the new [PROJECT]'s definition, not a structural relation. Extension is reserved for outer-scope additions only — a [PROJECT] needing finer resolution, different bin semantics, or a different shape is authored as a new [PROJECT] from scratch, not as an extension.
- Future [PROJECT] instantiations (legal, marketing, etc.) MAY declare different block counts, different segment functions, and different controlled vocabularies. The framework substrate accommodates this without modification; only the [PROJECT]'s [NODE_BLK] definitions change.
- A portable-block library and compiled-block packaging — by which [NODE_BLK] specifications could be shared across [PROJECT] instantiations — are reserved for v2.0+. v1.0 documents the structural commitment but does not ship the library.
- A future framework-level prefix block above block 1 (FROID/USERID/LICENSE) is reserved for v2.0+ to bridge X.660 addressing and per-tenant company namespaces. The new layout makes that prefix block easy to add cleanly above the current block 1.
