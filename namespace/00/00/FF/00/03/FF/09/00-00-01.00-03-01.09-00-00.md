## Definition

[FLIO_CRT] is the field-in-manifest that records the [NODE_ID] of the [PROJ_TIM] moment-folio at which this [FOLIO] was committed.

The manifest field name is `flio_crt`. Its value is a [NODE_ID] resolving to a [PROJ_TIM] moment in [FROID_NS] — second-resolution time addressing per [PROJ_TIM]'s 4-block layout.

[FLIO_CRT] has no [FLO_CRT] value class. Validation is reference-resolution; the referenced coordinate is a valid [PROJ_TIM] [NODE_ID] whether or not a moment-folio is materialized at that position.

## Rules

- `flio_crt`, when present, MUST be a [NODE_ID] in [PROJ_TIM] address-form (4-block layout with CC=02).
- The [PROJ_TIM] coordinate referenced by `flio_crt` is the commit-second of this [FOLIO] and is fixed thereafter.
- Substrate folios use the canonical reference `00-00-00.01-02-00.00-00-01.00-00-00` (epoch 0 / 1970-01-01 00:00:00 UTC under FROID_NS's RR=01 PROJ_TIM-of-substrate slot) when real commit timestamps have not been migrated from git history.

## Constraints

- `flio_crt` carries exactly one [NODE_ID].
- The referenced [PROJ_TIM] [NODE_ID] does NOT need to be materialized as a folio file for the reference to be valid; consistent with [FOLIO]'s rule that a valid [NODE_ID] position does not by itself imply that a [FOLIO] exists there.

## Relations

- [FLIO_CRT] is recorded by [FLIO_REF].
- [FLIO_CRT] references one [PROJ_TIM] moment by [NODE_ID].
- [FLIO_CRT] does NOT enumerate over a closed value class; it resolves by reference.

## Notes

- Aggregation by date (e.g., "all folios committed on 2026-05-05") is a derived view over [FLIO_CRT] values across the namespace, not a property recorded on individual folios. The future [PROJ_DAT] projection (deferred to v2.0+) provides the date-resolution addressing primitive; until then, date-bucketing is a build-artifact concern.
